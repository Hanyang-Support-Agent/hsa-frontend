## **1. 문서 개요**

### **1-1. 문서 목적**

본 문서는 고객문의 분류/답변 초안 에이전트의 PoC 구현 범위를 기준으로, 실제 개발 대상 기능을 정의하기 위해 작성한다.

PoC에서는 전체 서비스 비전 중 핵심 흐름이 동작하는 수준까지를 목표로 하며, 우선적으로 다음 기능을 검증한다.

- 문의 수집
- 문의 자동 분류
- DB 기반 자동응답
- RAG 기반 답변 초안 생성
- 관리자 검토 및 수정
- 응답 전송
- 로그 저장 및 조회

### **1-2. PoC 대상 범위**

- 문의 채널: **카카오톡, 인스타그램, 이메일**
- 문의 유형: **배송 문의, 교환/환불 문의, 상품 문의, 기타 문의**
- 데이터 범위:
    - 주문/배송 테스트용 DB
    - 상품 정보 문서
    - 환불/배송 정책 문서
    - FAQ 문서
- 관리자 기능:
    - 문의 목록 조회
    - 문의 상세 조회
    - 답변 초안 확인
    - 수정 및 발송
    - 로그 확인

# **3. 상세 기능명세**

## **3-1. 문의 수집 기능**

### **기능 ID**

F-01

### **기능명**

문의 수집 및 저장

### **기능 목적**

카카오톡, 인스타그램 및 이메일로 들어온 고객 문의를 시스템에 저장하고, 이후 분류 및 응답 프로세스의 입력값으로 사용한다.

### **기능 설명**

- 고객이 카카오톡, 인스타그램, 또는 이메일을 통해 문의를 남긴다.
- 시스템은 문의 내용을 통합된 데이터 구조로 저장한다.
- 저장 시 유입 채널, 고객 식별 정보, 문의 원문, 접수 시각 등을 함께 기록한다.

### **입력값**

- 유입 채널
    - 카카오톡
    - 인스타그램
    - 이메일
- 고객 식별 정보
    - 이름 또는 이메일 주소
- 문의 본문
- 접수 시각

### **출력값**

- 저장된 문의 ID
- 문의 원문
- 채널 정보
- 접수 상태(예: 접수 완료)

### **예외사항**

- 문의 본문이 비어 있는 경우 저장 불가
- 이메일 형식이 잘못된 경우 웹 문의에서 저장 제한 가능

### **우선순위**

상

## **3-2. 문의 자동 분류 기능**

### **기능 ID**

F-02

### **기능명**

고객 문의 자동 분류

### **기능 목적**

고객 문의를 유형별로 자동 분류하여 이후 처리 방식(자동응답/초안생성)을 결정할 수 있도록 한다.

### **기능 설명**

- 저장된 문의 내용을 AI가 분석한다.
- 문의는 아래 4개 유형 중 하나로 분류한다.
    - 배송 문의
    - 교환/환불 문의
    - 상품 문의
    - 기타 문의
- 분류 결과는 문의 데이터에 함께 저장된다.

### **입력값**

- 문의 본문

### **출력값**

- 문의 유형 분류 결과
- 분류 시각

### **분류 기준 예시**

- 배송 문의: 배송 현황, 송장, 도착 예정일 등
- 교환/환불 문의: 교환/환불 가능 여부, 교환/환불 절차, 교환/반품 요청 등
- 상품 문의: 원단, 사이즈, 색상, 세탁 방법 등
- 기타 문의: 위 유형으로 분류되지 않는 문의

### **예외사항**

- 복합 문의의 경우 기타문의로 처리

### **우선순위**

상

## **3-3. 자동응답 대상 판별 기능**

### **기능 ID**

F-03

### **기능명**

자동응답 가능 여부 판별

### **기능 목적**

문의가 DB 조회만으로 응답 가능한지 판단하여 자동응답 처리 여부를 결정한다.

### **기능 설명**

- 문의 분류 결과와 문의 내용을 바탕으로 자동응답 가능 여부를 판별한다.
- DB 조회만으로 답변 가능한 문의만 자동응답 대상으로 처리한다.
- 그 외 문의는 답변 초안 생성 대상으로 전환한다.

### **자동응답 대상 예시**

- 배송 상태 조회
- 주문 상태 조회
- 송장번호 안내
- 예상 도착일 확인

### **초안 생성 대상 예시**

- 환불 가능 여부 판단
- 상품 원단/세탁/사이즈 설명
- 정책 해석이 필요한 문의
- 예외 상황이 포함된 문의

### **입력값**

- 문의 유형
- 문의 내용
- 주문번호 또는 고객 식별 정보(존재 시)

### **출력값**

- 자동응답 가능 여부
    - 가능
    - 불가

### **우선순위**

상

## **3-4. DB 기반 자동응답 기능**

### **기능 ID**

F-04

### **기능명**

주문/배송 DB 조회 기반 자동응답

### **기능 목적**

주문/배송 상태처럼 DB 조회만으로 확인 가능한 문의에 대해 자동으로 응답을 생성하고 발송한다.

### **기능 설명**

- 시스템은 문의 내용에서 주문 관련 식별값을 추출하거나 고객 식별 정보를 사용해 DB를 조회한다.
- 조회 결과를 바탕으로 미리 정의된 응답 템플릿에 값을 삽입해 답변을 생성한다.
- 생성된 응답은 고객에게 자동 발송된다.

### **입력값**

- 문의 ID
- 주문번호 또는 고객 식별 정보
- 배송/주문 테스트용 DB

### **출력값**

- 자동 생성된 응답 문구
- 발송 결과
- 자동응답 여부 기록

### **응답 예시**

- “현재 고객님의 주문은 배송 중이며, 예상 도착일은 00월 00일입니다.”
- “주문하신 상품은 현재 상품 준비 중 상태입니다.”

### **예외사항**

- 주문 정보 조회 실패 시 자동응답 불가 처리
- 동일 고객의 주문이 여러 건일 경우 관리자 확인 대상으로 전환 가능

### **우선순위**

상

## **3-5. 문서 기반 답변 초안 생성 기능**

### **기능 ID**

F-05

### **기능명**

RAG 기반 답변 초안 생성

### **기능 목적**

DB 조회만으로 답변할 수 없는 문의에 대해, 정책 문서 및 상품 정보를 검색하여 운영자용 답변 초안을 생성한다.

### **기능 설명**

- 시스템은 문의 내용을 바탕으로 관련 문서를 검색한다.
- 검색 대상 문서는 다음과 같다.
    - 환불/교환 정책
    - 배송 정책
    - FAQ
    - 상품 정보 문서
- 검색된 문서를 기반으로 고객 문의에 대한 답변 초안을 생성한다.
- 생성 결과는 자동 발송하지 않고 관리자에게 제시한다.

### **입력값**

- 문의 본문
- 문의 유형
- 정책/FAQ/상품 문서

### **출력값**

- 답변 초안
- 참고 문서 또는 근거 정보
- 초안 생성 시각

### **예외사항**

- 관련 문서를 찾지 못한 경우 기본 초안 또는 “직접 응답 필요” 상태로 표시
- 문서 간 내용 충돌 시 운영자 검토 필요 표시

### **우선순위**

상

- 자동응답 여부 확인
- 답변 초안 확인
- 답변 수정 및 발송

### **로그 조회 화면**

- 처리 완료된 문의 이력 확인

### **문서 관리 화면**

- 문서 업로드
- 문서 목록 확인
- 문서 수정/삭제

# **5. 우선순위 정리**

PoC에서는 전체 서비스 비전을 모두 구현하기보다, **핵심 흐름이 실제로 동작하는지 확인할 수 있는 수준**까지를 목표로 한다.

따라서 기능은 **반드시 구현할 기능**과 **여유가 있을 경우 확장 가능한 기능**으로 구분한다.

## **5-1. 반드시 구현할 기능**

### **1) 문의 수집 및 저장**

- 카카오톡, 인스타그램 및 이메일을 통해 들어온 고객 문의를 수집
- 문의 원문, 고객 정보, 유입 채널, 접수 시각 등을 저장
- 이후 분류 및 응답 프로세스의 입력값으로 사용

### **2) 고객 문의 자동 분류**

- 고객 문의를 자동으로 분석하여 유형 분류
- PoC에서는 아래 4개 유형을 우선 지원
    - 배송 문의
    - 교환/환불 문의
    - 상품 문의
    - 기타 문의

### **3) 자동응답 가능 여부 판별**

- 분류된 문의 중에서 **DB 조회만으로 답변 가능한 문의인지 여부**를 판단
- 주문 상태, 배송 상태, 송장 정보 등 정형 데이터만으로 응답 가능한 경우 자동응답 대상으로 처리
- 그 외는 답변 초안 생성 대상으로 처리

### **4) DB 기반 자동응답**

- 주문/배송 테스트용 DB를 조회하여 자동응답 생성
- 예:
    - 배송 중 여부
    - 예상 도착일
    - 송장 번호
    - 주문 상태
- 조회 결과를 템플릿에 반영하여 자동응답 문구 생성

### **5) RAG 기반 답변 초안 생성**

- DB 조회만으로 답변이 어려운 문의에 대해 답변 초안 생성
- 환불 정책, 배송 정책, 상품 정보, FAQ 문서 등을 검색하여 초안 작성
- 생성된 초안은 고객에게 바로 발송하지 않고 관리자에게 제시

### **6) 관리자 문의 목록 및 상세 조회**

- 관리자가 전체 문의 목록을 확인할 수 있어야 함
- 문의별로 다음 정보를 조회 가능해야 함
    - 문의 원문
    - 유입 채널
    - 문의 분류 결과
    - 자동응답 여부
    - 생성된 답변 초안
    - 처리 상태

### **7) 관리자 응답 수정 및 발송**

- 관리자가 AI 초안을 검토하고 수정 가능
- 초안이 적절하지 않을 경우 직접 새 답변 작성 가능
- 최종적으로 관리자 확인 후 발송 처리

### **8) 로그 저장**

- 문의 접수부터 최종 응답까지의 전 과정을 기록
- 기록 예시:
    - 문의 원문
    - 문의 유형
    - 자동응답 여부
    - 생성된 초안
    - 관리자 수정 내용
    - 최종 발송 내용
    - 발송 시각

# **6. 비기능 요구사항**

비기능 요구사항은 시스템이 어떤 방식으로 동작해야 하는지를 정의하는 항목이다.

PoC에서는 아래 요구사항을 중심으로 고려한다.

## **6-1. 사용성**

- 관리자는 문의 목록과 처리 상태를 직관적으로 확인할 수 있어야 한다.
- 답변 초안 확인, 수정, 발송 과정이 복잡하지 않아야 한다.
- 화면은 짧은 학습만으로도 사용할 수 있을 정도로 단순해야 한다.

## **6-2. 정확성**

- DB 기반 자동응답은 주문/배송 테스트용 DB의 값을 정확히 반영해야 한다.
- 문서 기반 답변 초안은 업로드된 정책/상품 정보에 기반해야 한다.
- 잘못된 정보로 자동응답하지 않도록 자동응답 대상은 DB 조회가 명확한 경우로 제한한다.

## **6-3. 추적 가능성**

- 문의가 어떤 방식으로 처리되었는지 이력 확인이 가능해야 한다.
- 자동응답인지, 관리자 수정 후 발송인지 구분되어야 한다.
- 문의별 처리 흐름을 로그로 남겨야 한다.

## **6-4. 확장성**

- PoC 이후 문의 채널을 추가할 수 있어야 한다.
- 문의 유형을 확장할 수 있어야 한다.
- 상품 정보 및 정책 문서 종류를 늘릴 수 있어야 한다.
- 향후 다양한 쇼핑몰 플랫폼과 연동할 수 있도록 구조를 고려해야 한다.

## **6-5. 안정성**

- 문의 데이터와 응답 데이터는 누락 없이 저장되어야 한다.
- 문의 처리 중 오류가 발생해도 최소한 원문과 처리 상태는 확인 가능해야 한다.

# **9. PoC 범위 내 구현 시 유의사항**

## **9-1. 자동응답 범위 제한**

PoC에서는 모든 문의를 자동응답 대상으로 삼지 않는다.

자동응답은 반드시 **DB 조회만으로 사실 확인이 가능한 문의**에 한정한다.

예:

- 배송 상태
- 송장 정보
- 주문 상태

이렇게 범위를 제한함으로써 잘못된 자동응답의 위험을 줄일 수 있다.

## **9-2. 초안 생성은 보조 기능으로 사용**

RAG 기반 답변 생성은 **최종 응답 자동화**가 아니라 **운영자 응답 보조**를 목적으로 한다.

즉, 정책 해석이나 상품 설명이 필요한 문의는 초안을 제공하되, 최종 발송은 관리자가 수행하도록 한다.

## **9-3. 문서 관리 기능은 최소 구현**

문서 업로드 기능은 포함하되, 수정/삭제/버전 관리까지 모두 정교하게 구현하기보다

우선은 **답변 초안 생성을 위한 문서 등록 기능** 중심으로 구현한다.
